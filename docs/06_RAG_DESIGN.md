# ACOS RAG Design

## Architecture Overview

```
User Query / Generation Request
         │
         ▼
  Intent Classification
         │
         ▼
  Query Embedding (nomic-embed-text via Ollama)
         │
         ▼
  Collection Routing (select relevant ChromaDB collections)
         │
         ▼
  Vector Search (top-K per collection)
         │
         ▼
  Metadata Filtering (confidence, date, source_type)
         │
         ▼
  Cross-Collection Merge & Deduplication
         │
         ▼
  Re-ranking (BM25 keyword + semantic score blend)
         │
         ▼
  Context Window Assembly (token budget management)
         │
         ▼
  LLM Generation (Ollama / Qwen3 8B)
         │
         ▼
  Confidence Assignment
         │
         ▼
  Response + Evidence Panel
```

---

## Embedding Model

**Model:** `nomic-embed-text` (via Ollama)  
**Dimensions:** 768  
**Reason:** Runs fully locally, strong retrieval quality, small footprint.

All embeddings generated by `backend/rag/embedder.py` which wraps the Ollama `/api/embeddings` endpoint.

## Chunking Strategies

### Experience Bullets
- **Unit:** One ChromaDB document = one `experience_bullet` row
- **Content:** `{title} at {company} ({start_date}–{end_date}): {bullet_text}`
- **Rationale:** Bullets are already atomic units of evidence; no further splitting needed

### Projects
- **Unit:** One ChromaDB document = one project record
- **Content:** `Project: {name}\nDescription: {description}\nTech: {tech_stack}\nIndustry: {industry}`
- **Rationale:** Projects are short enough to keep unified; splitting loses context

### Cover Letters
- **Unit:** Paragraph-level chunks, max 300 words
- **Content:** Raw paragraph text
- **Rationale:** Voice patterns emerge at paragraph level, not sentence level

### Job Descriptions
- **Unit:** Section-level chunks (Requirements, Responsibilities, About Company)
- **Detection:** Regex pattern matching on common JD section headers
- **Max chunk size:** 500 words, with 50-word overlap between chunks
- **Rationale:** Keyword extraction needs full requirement context; splitting by section preserves logical coherence

### GitHub Repositories
- **Unit:** Per-file chunks for README, core source files; max 500 words
- **Overlap:** 50 words
- **Filter:** Skip binary files, lock files, generated files
- **Rationale:** README is highest signal for project/skill extraction; source files provide tech stack evidence

### Claude Exports
- **Unit:** Per-project, per-conversation chunk; max 400 words
- **Content:** Project name + conversation context
- **Rationale:** Conversation excerpts reveal reasoning skills and architecture patterns

## Retrieval Parameters

```python
# Default parameters (tunable via system_config)
TOP_K_PER_COLLECTION = 10
FINAL_CONTEXT_DOCS = 15
MIN_SIMILARITY_THRESHOLD = 0.35
RERANK_BM25_WEIGHT = 0.3
RERANK_SEMANTIC_WEIGHT = 0.7
MAX_CONTEXT_TOKENS = 4096   # leaves ~4k for generation with Qwen3 8B 8k window
```

---

## Re-ranking Algorithm

```
1. Collect top-K candidates from each collection
2. For each candidate, compute:
   score = (semantic_similarity * 0.7) + (bm25_keyword_score * 0.3)
3. Apply multipliers:
   × 1.2 if confidence_level == 'verified'
   × 1.0 if confidence_level == 'strong_inference'
   × 0.6 if confidence_level == 'weak_inference'
4. Apply outcome signal boost (from outcome_signals table):
   × (1 + log(1 + experience_outcome_weight))
5. Deduplicate by source entity (keep highest-scoring chunk per entity)
6. Select top FINAL_CONTEXT_DOCS by combined score
7. Return ranked list with metadata for evidence panel
```

## Context Window Assembly

Token budget for Qwen3 8B (8K context):

```
System prompt:          ~400 tokens
User query + intent:    ~200 tokens
Retrieved context:     ~4096 tokens   ← capped here
Output generation:     ~3000 tokens
Safety margin:          ~304 tokens
Total:                  8000 tokens
```

Assembly order (most relevant first):
1. Verified evidence chunks
2. Strong inference chunks
3. Weak inference chunks (only if budget remains)

## ChromaDB Client Configuration

```python
# backend/rag/chroma_client.py
import chromadb

client = chromadb.PersistentClient(path=CHROMA_DB_PATH)
```

Collections are created idempotently at startup. No external ChromaDB server required.
